
# AI Developer Session Log: O2C Graph & NL-Query System
**Project:** Forward Deployed Engineer Assessment - SAP Data Visualization
**Models Used:** Gemini 3 Pro (Reasoning/Chat), Claude Haiku 4.5 (UI), GPT 5.1 Codex Max (Backend/SQL)

### **Note on Session Data**
The following is a condensed summary of my development sessions. My exported chat history for this project totaled **140MB** in JSON format, largely due to the high volume of numerical data, repeated code blocks, and deep-dive technical explanations generated by the models. To remain concise, I am summarizing the key technical pivots and workflows here.

---

## **Phase 1: Data Ingestion & Schema Mapping**
**The Goal:** Load SAP Order-to-Cash (O2C) JSONL files into a queryable format.

*   **How we started:** I initially used **GPT 5.1 Codex Max** to help write a Python ingestion script. The raw dataset wasn't massive, but the SAP field names (like `A_SalesOrder`, `SoldToParty`) were cryptic.
*   **The Conflict:** The AI originally suggested a standard relational mapping, but I realized that to make the "Chat" feature work, the LLM needed a very clear "Data Dictionary." 
*   **The Decision:** We decided to use **DuckDB** as the engine. It allowed us to keep the original SAP naming conventions while providing a high-performance SQL interface. I prompted the AI to generate a `schema_map.py` that translated the technical SAP keys into human-readable descriptions for the LLM to use during inference.

---

## **Phase 2: UI Evolution (The Pivot from Sigma to 3D)**
**The Goal:** Create a graph visualization that makes sense of document flows.

*   **How we started:** I used **Claude Haiku 4.5** to build a 2D graph using **Sigma.js**. 
*   **The Conflict:** As soon as we started rendering the O2C flows, the 2D layout became a "hairball." Because one Sales Order can link to multiple Deliveries and Billing documents, the nodes overlapped, making it impossible to see the "Broken Flows" (deliveries without invoices). We also kept hitting CSS container height issues where Sigma wouldn't mount correctly.
*   **The Resolution:** We made the executive decision to pivot to **`react-force-graph-3d`**. This wasn't just for "looks"—the 3D perspective allowed me to use the Z-axis to separate the stages of the supply chain (e.g., Orders at the bottom, Payments at the top). This immediately made the data "scannable."

---

## **Phase 3: NL-to-SQL Pipeline & Logic Guardrails**
**The Goal:** Let users ask questions like "Which orders are stuck?" and get a natural response.

*   **How we started:** I used **Gemini 3 Pro** to build the chat pipeline. The initial prompt was too "helpful"—it tried to answer general questions about SAP instead of staying focused on my specific dataset.
*   **The Conflict:** The LLM would sometimes hallucinate column names that didn't exist in my DuckDB tables, causing the backend to crash.
*   **The Decision:** I implemented a **"Strict Grounding"** workflow. I provided Gemini with the exact DuckDB schema and a few-shot prompt. I told the AI: *"If the SQL you generate doesn't use these specific tables, or if the user asks something off-topic, trigger a fallback message."*
*   **Refinement:** I had to manually intervene when the AI struggled with the "Broken Flow" logic. I guided it to write a specific `LEFT JOIN` in DuckDB to identify orphaned delivery IDs, which we then highlighted in **Red** in the 3D graph.

---

## **Technical Summary of Decisions**
*   **Backend:** Switched to a **DuckDB + FastAPI** stack for local speed and simplicity.
*   **Frontend:** Pivoted from **Sigma.js (2D)** to **react-force-graph-3d** to solve visual congestion and document-traceability issues.
*   **AI Strategy:** Used **Gemini** for its reasoning and conversational "narration" of data, while relying on **GPT Codex** for the heavy-lifting SQL generation.
*   **Context Management:** Managed the massive 140MB chat history by extracting key "Logic Blocks" to ensure the AI stayed focused on the current implementation state.
